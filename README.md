# Deribit Price Tracker

Сервис для отслеживания 
индексных цен BTC и ETH с биржи Deribit. 
Каждую минуту Celery-таска забирает 
актуальные цены через асинхронный HTTP-клиент и сохраняет их в PostgreSQL. 
Данные доступны через REST API.

Проект выполнен в рамках тестового задания. [Текст тестового задания.](./docs/task.png)

## API

Все эндпоинты принимают обязательный query-параметр `ticker`.

### GET /api/v1/prices/

Возвращает все сохранённые цены по указанному тикеру, отсортированные по убыванию (сначала новые).
```
GET /api/v1/prices/?ticker=btc_usd
```

Опциональная фильтрация по дате. Дата передаётся в формате ISO с обязательным указанием timezone:
```
GET /api/v1/prices/?ticker=btc_usd&date_from=2026-03-14T00:00:00+00:00&date_to=2026-03-14T23:59:59+00:00
```

### GET /api/v1/prices/latest

Возвращает последнюю сохранённую цену по указанному тикеру.
```
GET /api/v1/prices/latest?ticker=btc_usd
```

### Пример ответа

```json
{
  "id": 1,
  "name": "btc_usd",
  "price": 70682.96,
  "timestamp": 1773507626
}
```

## Design Decisions

### **TickerService как класс**
Бизнес-логика работы с ценами инкапсулирована в классе TickerService. 
Сессия БД передаётся через конструктор — класс не знает откуда она берётся и не создаёт её сам. Это следует принципу dependency injection и делает каждый слой приложения независимым от деталей реализации других слоёв.

### **DeribitClient как асинхронный контекстный менеджер**
Клиент реализован через `__aenter__`/`__aexit__`. 
Это гарантирует закрытие `aiohttp.ClientSession` 
даже при возникновении исключений, и инкапсулирует 
управление жизненным циклом HTTP-сессии внутри класса.

### **Celery + RabbitMQ для фоновых задач**  
Для периодического получения цен используется 
Celery с брокером RabbitMQ. Celery Beat запускает 
таску fetch_prices по расписанию раз в минуту, 
Celery Worker её выполняет — забирает цены с 
Deribit и сохраняет в БД. Такой подход полностью 
отделяет фоновую работу от API - веб-сервер не знает 
ничего о периодических задачах и не блокируется 
на время их выполнения.

### **Query параметры вынесены в Depends**
Параметры фильтрации сгруппированы 
в `PriceFilterParams` и инжектируются через 
`Depends(get_price_filters)`. Это отделяет 
валидацию от роутера, делает фильтры 
переиспользуемыми и позволяет тестировать 
валидацию независимо от HTTP-слоя.

### **Единый эндпоинт для фильтрации**
`GET /prices/` покрывает два требования — получение 
всех данных и фильтрацию по дате через опциональные 
параметры `date_from` и `date_to`. Разделять их в два 
отдельных роута избыточно, так как это один и тот же 
ресурс с опциональной фильтрацией.

### **AwareDatetime для фильтрации по дате**
Вместо `datetime` используется `AwareDatetime` -
Pydantic тип который требует явного указания timezone 
от клиента. Это предотвращает неоднозначную 
интерпретацию времени на серверах в разных часовых 
поясах и делает контракт API явным.

### **Отдельный контейнер для тестовой БД**
Тестовая БД вынесена в отдельный сервис 
`db_test` в `docker-compose.yml`. 
Это позволяет запускать тесты не затрагивая 
основную БД приложения.
